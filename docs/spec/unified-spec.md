## 1. 产品概述

### 1.1 核心理念

“Every idea counts”——每个想法都值得被认真对待，不应只停留在备忘录里被遗忘。

### 1.2 目标用户

- 创业者：验证商业想法，快速原型思考
- 创作者：捕捉创作灵感，深化创意概念
- 学生：记录学习洞察，结构化知识体系

### 1.3 核心价值

- 从捕获到孵化：不只是记录，而是让想法生长
- AI 协作补足：填补用户思维盲区，提出关键问题
- 结构化输出：将散乱想法转化为可行动的笔记

### 1.4 关键差异化（定位）

- “对话只是过程，笔记才是产物”：AI 对话的目标是产出结构化、可行动的笔记
- “状态机驱动的孵化”而不是泛聊天：严格控制对话推进，避免跑题与空泛

## 2. 关键用户旅程（MVP 用户路径）

### 2.1 快速捕获 → 进入孵化

1) 用户通过快捷入口打开快速捕获面板
2) 选择想法类型（产品/创业、创作灵感、学习研究）
3) 输入一句话想法（支持基础 Markdown）
4) 草稿在本地自动缓存（未提交也不丢）
5) 提交后创建 `idea` 记录（含 `idea_type`），状态为 `draft`
6) 进入孵化模式（状态切换为 `incubating`）

### 2.2 孵化对话 → 生成结构化笔记

1) 用户与 AI 多轮问答
2) 允许跳过问题、用户主动提问、随时结束
3) 结束后触发“结构化提取”，生成最终笔记
4) 用户可编辑笔记并保存，状态进入 `completed` 或继续孵化

### 2.3 想法库管理 → 检索/归档/导出

1) 列表卡片展示全部想法
2) 按状态筛选、全文搜索、标签筛选、排序
3) 批量归档/删除
4) 导出（Markdown/PDF/HTML/Notion）或分享（链接/图片/邮件）

## 3. 功能规格（Functional Specification）

### 3.1 快速捕获（Quick Capture）

- 输入方式：文本输入（支持 Markdown 基础语法）
- 想法类型：在捕获时手动选择（`product`/`creative`/`research`）
- 快捷入口（PWA MVP）：
  - 应用内快捷键（仅在页面聚焦时生效）
  - 移动端：添加到主屏幕后，从图标快速进入捕获页（可选：系统分享面板“分享到 Every Idea Counts”作为输入入口）
- 快捷入口（可选增强，非 PWA 原生能力）：
  - 全局快捷键（`Cmd+Shift+I` / `Ctrl+Shift+I`）需要通过“桌面壳（如 Tauri）”或“浏览器扩展”实现
- 极简界面：单输入框 + 提交按钮
- 智能提示文案：例如“用一句话描述你的想法”“你遇到了什么问题？”
- 自动保存：未提交草稿本地缓存（断电/崩溃后可恢复）

### 3.2 AI 对话孵化（AI Incubation Chat）

- 对话模式：多轮问答式交互
- 会话长度（MVP）：默认只推进 1 个阶段（1 个问题 + 最多 1 次追问）；用户选择“继续孵化”才进入下一阶段
- AI 角色：引导者 + 思考伙伴 + 结构化助手
- 对话控制：可跳过问题、可主动提问、可随时结束

用户主动提问（Ask AI，MVP 必做）：
- 在孵化输入框上方提供 `Answer` / `Ask AI` 切换
- Ask AI 的回答：最多 3 句；不反问；不推进阶段；不写入 collected；回答后自动切回 Answer 并重新展示当前阶段主问题

核心问题库（固定大纲，动态追问）：
- 这个想法要解决什么核心问题？
- 目标用户是谁？他们有什么痛点？
- 与现有解决方案相比，独特价值是什么？
- 实现这个想法需要哪些关键步骤？
- 可能遇到的最大挑战/风险是什么？

动态追问要求：根据用户回答智能调整后续问题；优先让用户“具体化”（更具体的对象、场景、约束、可验证假设）。

按想法类型（`idea_type`）的提问角度差异（共享 5 阶段状态机，但每一阶段的“示例追问/行动项生成规则”不同）：
- `product`：强调目标用户、替代方案、差异化、最小验证实验、风险
- `creative`：强调受众与表达形式、结构大纲、素材缺口、发布节奏
- `research`：强调概念边界、反例、可验证实验、阅读清单、应用场景

### 3.3 笔记生成与编辑（Note Generation & Editing）

- 自动生成：基于对话内容生成结构化笔记
- 笔记模板：采用固定 Markdown 结构，生成后可编辑
- 编辑能力（MVP）：Markdown 编辑 + 实时预览
- 编辑能力（后续增强）：富文本编辑器（类 Notion 体验）

统一笔记模板（融合版，默认采用 v1.0 的 Emoji 分段以增强情绪与识别度）：

```markdown
# [AI 生成的标题]

## 💡 核心概念
[1-2 句话总结核心理念]

## 🎯 解决的问题
[具体描述要解决的痛点]

## 👥 目标用户
[用户画像描述]

## ✨ 独特价值
[与竞品的差异化]

## 🔍 关键洞察
[对话中产生的深度思考]

## ❓ 待探索问题
[需要进一步验证的假设]

## 📋 行动项
- [ ] 具体可执行的下一步
- [ ] 需要调研的信息
- [ ] 可联系的相关人员

---

*创建时间：{timestamp}*
*标签：{auto-generated tags}*
```

### 3.4 想法库管理（Idea Library）

- 列表视图：卡片式展示所有想法
- 状态筛选：`draft` / `incubating` / `completed` / `archived`
- 搜索：全文搜索 + 标签筛选
- 排序：创建时间、更新时间、标题
- 批量操作：批量删除、批量归档
- 回顾机制（MVP）：每周回顾视图（聚合 `draft/incubating`，提供“继续孵化/生成笔记/归档”的快捷动作）
- 回顾排序（MVP）：默认按“最久未更新（updated_at 最早）”优先

### 3.5 标签系统（Tagging System）

- 自动标签：AI 根据内容生成相关标签
- 手动标签：用户可添加自定义标签
- 标签管理：查看所有标签；支持合并相似标签

### 3.6 导出与分享（Export & Share）

- 导出格式（MVP）：Markdown、PDF、HTML
- 分享选项：复制链接、生成分享图片、发送到邮箱

Notion 相关（MVP）：仅提供 Markdown/HTML 导出，便于用户手动导入 Notion；不做 API 直连同步。

## 4. AI Prompt 引擎架构（融合自 v1.1）

目标：确保 AI 稳定扮演“引导者 + 思考伙伴”，避免多轮对话偏离主题；将对话推进过程工程化。

### 4.1 状态机代理流（State Machine Agent Flow）

系统维护 `IncubationState`（对话进度对象），严格追踪五个核心问题状态：

- `problem_definition`
- `target_audience`
- `unique_value`
- `execution_steps`
- `risk_assessment`

### 4.2 三层 Prompt 架构

Layer 1：全局系统设定（System Prompt）
- 设定 AI 人设与“红线约束”
- 语气要求：专业、简练
- 交互要求：绝对不要一次性问多个问题；更多追问；少总结、先澄清

Layer 2：动态状态控制（Dynamic Context & State Prompt / 路由控制器）
- 输入：用户初始想法、当前状态、该状态目标、用户最新回复
- 输入（融合补充）：想法类型 `idea_type`（决定追问角度与行动项生成规则）
- 输出：只允许执行一个动作：
  - 分支 A：目标未达成 → 提出一个具体犀利的追问
  - 分支 B：目标达成 → 简短肯定并推进到下一阶段问题

参考路由控制模板（融合保留）：

```text
[基础上下文]
用户的初始想法是：“{raw_input}”
我们当前正处于想法孵化的第 {current_step_index} 步：【{current_state_name}】。
本阶段的核心目标是弄清楚：{current_state_goal}

[评估与行动指南]
请仔细分析用户刚刚的回复：“{user_latest_reply}”。
你必须基于以下逻辑进行且仅进行一项操作：

分支 A：目标未达成（需要追问）
判断标准：用户的回答过于宽泛、模糊。
你的行动：提出一个具体、犀利的追问，促使深入思考。

分支 B：目标已达成（推进状态）
判断标准：用户清晰地回答了核心目标。
你的行动：简短肯定，然后直接抛出下一个阶段的问题：“{next_state_question}”。
```

Layer 3：结构化提取（Data Extraction Prompt）
- 触发时机：孵化结束时
- 行为：切断对话流（不再继续聊天），用低随机性设置，将对话映射到固定 Markdown 模板

## 5. 技术架构

### 5.1 整体架构

```
Frontend (Web/PWA + React)  →  Backend (Supabase)  →  Database (PostgreSQL)
              │
              └────────────→  AI APIs (GLM / Qwen)
```

安全建议（融合补充）：
- 默认通过后端代理调用 AI（例如 Supabase Edge Functions/Serverless），避免在前端暴露供应商密钥
- MVP 采用“你自己的 API Key”（单一 Key，服务端环境变量保存）；不在前端持久化 Key
- 若未来扩展到“每个用户自带 Key（BYOK）”，可允许前端直连或服务端代管，但需要明确告知风险与存储策略

### 5.2 技术栈

- 前端框架：React 18 + TypeScript
- Web 形态：PWA（Service Worker + Web App Manifest + 离线缓存/IndexedDB）
- 状态管理：Zustand
- UI：Tailwind CSS + Headless UI
- 后端：Supabase（含数据库 + 认证 + 存储）
- 数据库：PostgreSQL
- AI：GLM / Qwen API（双供应商，可切换）

兼容性路线（与原文档对齐）：
- 桌面端（Tauri）作为后续形态：用于“全局快捷键/系统级能力/更深离线能力”

### 5.3 数据流

捕获流程：`用户输入 → 本地缓存 → 提交 → 创建 ideas 记录 → 进入孵化模式`  
孵化流程：`用户消息 → 发送到 AI → AI 回复 → 展示对话 → 结束 → 生成最终笔记 → 更新 ideas 记录`

## 6. 数据模型

### 6.1 核心表结构：ideas

```sql
CREATE TABLE ideas (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  idea_type TEXT NOT NULL DEFAULT 'product'
    CHECK (idea_type IN ('product', 'creative', 'research')),
  title TEXT NOT NULL,
  raw_input TEXT NOT NULL,
  ai_conversation JSONB NOT NULL DEFAULT '[]',
  final_note TEXT,
  tags TEXT[] DEFAULT '{}',
  status TEXT NOT NULL DEFAULT 'draft'
    CHECK (status IN ('draft', 'incubating', 'completed', 'archived')),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

### 6.2 状态机（ideas.status）

```text
draft ──[开始孵化]──▶ incubating ──[生成笔记]──▶ completed
  │                                           │
  └──[直接归档]───────────────────────────────┼──[归档]──▶ archived
                                              │
                                              └──[重新孵化]──▶ incubating
```

## 12. 风险与应对（去重融合）

技术风险：
- AI API 限制/不稳定：准备多个供应商（GLM/Qwen），实现快速切换容灾
- 性能问题：对话历史分页加载，避免内存溢出
- 数据丢失：自动备份 + 本地缓存双重保障

产品风险：
- 用户粘性不足：增加定期回顾提醒，形成使用习惯
- AI 质量不稳定：提供人工编辑能力，用户始终掌控
- 功能过于复杂：严格遵循 MVP 原则，逐步迭代

- 用户主动提问（MVP）：支持 Ask AI 模式（Answer/Ask AI 切换；不推进阶段）

## 17. 附录：AI 孵化 Prompt 规范（MVP）

本附录用于把“AI 引导”做成可控的状态机代理流，确保：不跑题、不空泛、一次只问一个问题、默认单阶段推进。

### 17.1 Prompt 架构（3 层）

Layer 1：System Prompt（全局人设与红线）
- 角色：专业、简练、带一点“犀利”的教练
- 硬规则：一次只问 1 个问题（不允许一条消息多个问号）；避免长篇总结；优先具体化；不迎合

Layer 2：State Router Prompt（动态路由控制器）
- 决策：`FOLLOW_UP`（追问）或 `ADVANCE`（推进）
- MVP 限制：每个阶段最多 2 轮用户回复（1 问 + 最多 1 追问）；第二轮仍模糊也要推进并标记待补充

Layer 3：Extraction Prompt（结构化提取）
- 将对话/阶段答案收敛映射到固定 Markdown 模板
- 行动项强制：至少 1 条“15 分钟内可做” + 至少 1 条“调研/验证”

### 17.2 对话状态（应用侧维护）

```json
{

  "raw_input": "string",
  "current_state": "problem_definition|target_audience|unique_value|execution_steps|risk_assessment",
  "current_step_index": 1,
  "turn_count_in_state": 0,
  "collected": {
    "problem_definition": null,

    "execution_steps": null,
    "risk_assessment": null
  },
  "note_regen_policy": "append",
  "language": "zh-CN"
}
```

### 17.3 Layer 1：System Prompt（通用模板）

```text
你是 Every Idea Counts 的“想法孵化教练”。

你的目标：把用户的一句话想法推进到“更具体、可验证、可行动”的最小方案。

硬性规则：
1) 一次只问 1 个问题（绝不允许一条消息里多个问号）。
2) 语言简练：优先 1-3 句；不要长篇总结。
3) 不迎合：当用户模糊时要追问，让用户具体化。
4) 不跑题：严格围绕当前阶段目标。
5) 不要输出 Markdown 笔记（除非进入“结构化提取”阶段）。

交互风格：专业、直接、友好，但更像教练而不是聊天机器人。
```

轨道补充（作为额外上下文注入）：
- product：强调用户、痛点、替代方案、差异化、最小验证实验。
- creative：强调受众、表达形式、结构、素材缺口、发布节奏。
- research：强调概念边界、反例、可验证实验、阅读清单、应用场景。

### 17.4 Layer 2：State Router Prompt（动态路由控制器）

输入模板：

```text
[基础上下文]
用户的初始想法是：“{raw_input}”
想法类型：{idea_type}
我们当前正处于想法孵化的第 {current_step_index} 步：【{current_state_name}】。
本阶段的核心目标是弄清楚：{current_state_goal}
本阶段已发生用户回复次数：{turn_count_in_state}（最多 2）

[用户最新回复]
“{user_latest_reply}”

[你的任务]
你必须基于以下逻辑进行且仅进行一项操作：

分支 A：目标未达成（需要追问）
判断标准：用户回答过于宽泛、缺少具体对象/场景/约束/可验证点。
行动：输出 FOLLOW_UP，并给出一个“更具体、单一”的追问。

分支 B：目标已达成（推进状态）
判断标准：用户回答能支撑下一步行动，已足够具体。
行动：输出 ADVANCE，并给出“下一阶段的单一问题”。

额外硬规则：
- 你输出的问题必须只有一个问号。
- 如果 turn_count_in_state 已经是 1（意味着这次是第二轮），即使仍不完美，也要优先 ADVANCE。
```

输出格式（严格）：

```json
{
  "action": "FOLLOW_UP|ADVANCE",
  "question": "string"
}
```

### 17.4.1 Ask AI 模式（用户主动提问，MVP）

目的：允许用户在不改变 stage 的前提下向 AI 询问概念/例子/建议，用于辅助回答当前阶段。

Ask AI 的输出规则（硬性）：
- 最多 3 句
- 不要反问（不要提出新问题）
- 不要总结当前阶段，也不要推进到下一阶段
- 不要输出 Markdown 笔记

Ask AI Prompt（示例，应用侧在 Ask AI 模式使用）：

```text
你是 Every Idea Counts 的孵化教练。

当前阶段：{current_state_name}
阶段目标：{current_state_goal}
想法类型：{idea_type}
原始想法：{raw_input}

用户现在是在 Ask AI 模式下提问，你的任务是直接回答用户的问题。

硬性规则：
1) 最多 3 句。
2) 不要反问。
3) 不要推进阶段，不要提出下一阶段问题。
4) 不要输出 Markdown 笔记。

用户问题：{user_question}
```

### 17.5 五阶段状态机：目标/主问题/追问/完成标准（按三轨道）

说明：每个阶段都必须输出“单一问题”。当模糊时追问一次，第二轮仍模糊也要推进。

#### 17.5.1 problem_definition（问题定义）

目标：把 raw_input 变成一句“为谁，在什么场景，解决什么痛点/达到什么结果”。

- product
  - 主问题：你希望这个想法为哪一类人解决什么具体问题？
  - 追问：若抽象（效率/体验/AI），追问“在什么流程/场景里、现在怎么做、最痛的点是什么”。
  - 完成：出现明确用户群体 + 场景 + 痛点/目标（至少 2/3）。

- creative
  - 主问题：这条灵感你想表达的核心一句话是什么？
  - 追问：若“还没想好”，追问“你希望受众看完产生什么感受或行动”。
  - 完成：一句话主题/观点可复述。

- research
  - 主问题：这个洞察的结论是什么，用一句话怎么说？
  - 追问：若“可能/也许”，追问“你想验证的命题是什么”。
  - 完成：命题清晰可验证/可证伪。

#### 17.5.2 target_audience（目标对象/受众）

目标：明确“第一优先对象”，以及他们的特征/动机（或应用场景）。

- product
  - 主问题：你最先要服务的那类用户是谁，他们现在用什么替代方案？
  - 追问：若泛（所有人/学生/年轻人），追问“一个具体人设：职业/频率/关键情境”。
  - 完成：具体画像 + 替代方案。

- creative
  - 主问题：你希望这条内容主要给谁看（一个具体受众画像）？
  - 追问：若泛，追问“你更想取悦哪一类人，而不是所有人”。
  - 完成：受众明确。

- research
  - 主问题：这个洞察主要用于解决谁的什么研究/工程问题？
  - 追问：若只是“学习”，追问“你希望把它用到哪一个项目/课题里”。
  - 完成：应用场景明确。

#### 17.5.3 unique_value（独特价值/差异化）

目标：形成“对比句”，明确为什么值得做。

- product
  - 主问题：相对现有方案，你的独特价值具体是什么（用一句对比句描述）？
  - 追问：若“更好/更智能”，追问“更好在哪个指标上，怎么验证”。
  - 完成：参照物 + 可验证差异点。

- creative
  - 主问题：这条内容最特别的角度/结构是什么，和常见内容有什么不同？
  - 追问：若不清，追问“你最想保留的一个独特点是什么”。
  - 完成：独特角度/表达方式清晰。

- research
  - 主问题：这个洞察相比常识或已有结论，新增价值在哪里？
  - 追问：若不清，追问“它改变了你接下来会怎么做”。
  - 完成：能表达“因此我会……”的决策变化。

#### 17.5.4 execution_steps（执行步骤/最小推进）

目标：落到 1 个最小实验或最小行动序列，并能产生证据/产出。

- product
  - 主问题：你下一周能做的最小验证步骤是什么（能产生证据的那种）？
  - 追问：若“大而全开发”，追问“一个不写代码也能验证的方式”。
  - 完成：至少 1 个步骤 + 预期证据。

- creative
  - 主问题：你下一步要产出的最小作品是什么（例如 300 字开头/5 个标题/一页大纲）？
  - 追问：若太大，追问“15 分钟内能完成的第一步”。
  - 完成：具体产出物 + 期限。

- research
  - 主问题：你下一步最小实验/阅读是什么，怎么判断学到了什么？
  - 追问：若泛读，追问“最先读哪一篇/复现哪一段”。
  - 完成：最小行动 + 判断标准。

#### 17.5.5 risk_assessment（风险/阻碍）

目标：识别最大阻碍并给出缓解动作。

- product
  - 主问题：这件事最大的风险/不确定性是什么，你怎么最小成本降低它？
  - 追问：若“时间不够”，追问“最先砍掉哪个假设/功能”。
  - 完成：最大风险 + 缓解动作。

- creative
  - 主问题：你最大的阻碍是什么（时间/素材/结构/表达），你能怎么绕开？
  - 追问：若阻碍多，追问“最先解决哪一个”。
  - 完成：一个阻碍 + 一个绕开方法。

- research
  - 主问题：你最可能卡住的点是什么（数据/数学/实现/理解），你怎么提前验证？
  - 追问：若不确定，追问“最难的一个概念是什么”。
  - 完成：一个卡点 + 一个验证手段。

### 17.6 Layer 3：结构化提取（Markdown Note Extraction）

提取 Prompt（示例）：

```text
你是结构化写作器。请把输入内容整理成 Markdown 笔记。

硬性规则：
1) 严格输出指定模板，不要添加额外小节。
2) 内容必须具体，避免空话。
3) 行动项至少两条：第一条必须 15 分钟内可完成；第二条必须是调研/验证。
4) 如果某些信息缺失，用“（待补充：……”占位，但占位必须指向一个明确问题。

[输入]
想法类型：{idea_type}
原始想法：{raw_input}
阶段答案：{collected_json}
```
